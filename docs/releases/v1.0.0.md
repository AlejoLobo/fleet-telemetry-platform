# Fleet Telemetry Platform 1.0.0

Notas formales de la primera versión estable y demostrable.

> El tag `v1.0.0` y el GitHub Release se crean **después** de fusionar
> `release/1.0.0` → `main` y de aprobar el CI en `main`.

## 1. Resumen ejecutivo

`1.0.0` consolida la vertical completa de monitoreo de flotas: ingesta HTTP,
mensajería Kafka, procesamiento at-least-once, persistencia TimescaleDB,
dashboard Next.js en tiempo real, app móvil offline-first, agente IA operativo
e infraestructura AWS **dev** reproducible. Es una versión demostrable y
operable en local/Compose; **no** es una declaración de alta disponibilidad
productiva.

## 2. Capacidades principales

| Área | Capacidad |
|------|-----------|
| Ingesta | `POST /api/telemetry` y batch → `202` |
| Mensajería | Topic `telemetry.raw`, key `DeviceId` |
| Procesamiento | Worker at-least-once, commit manual de offsets |
| Persistencia | TimescaleDB hypertable + idempotencia por `EventId` |
| Identidad | `DeviceId` + `VehicleName` + `VehicleType` |
| Tiempo real | SSE KafkaPush (polling solo como alternativa) |
| Dashboard | Next.js 15 + React 19, modos API y Demo |
| Mobile | Expo 54 + SQLite offline-first |
| IA | Tools operativas; OpenAI opcional / mock |
| Infra | Terraform AWS **dev** (EC2 + Compose + ALB) |
| Observabilidad | OpenTelemetry opt-in (sin collector empaquetado) |

## 3. Arquitectura entregada

```text
Mobile/HTTP → API (.NET 10) → Kafka/Redpanda → Worker
                                      ↓
                               TimescaleDB
                                      ↓
                    REST + SSE ← Dashboard (Next.js)
```

Clean Architecture en backend (Domain / Application / Infrastructure / Api / Worker).
Read model de flota, alertas con deduplicación/cooldown y migraciones de esquema
hasta v8 (`vehicle_type`).

## 4. Garantías técnicas

| Garantía | Estado en 1.0.0 |
|----------|-----------------|
| At-least-once (Kafka → persistencia) | Sí |
| Exactly-once end-to-end | No |
| Idempotencia por `EventId` | Sí |
| Commit de offset solo tras éxito o DLQ seguro | Sí |
| DLQ para payloads inválidos / fallos de contrato agotados | Sí |
| Errores inesperados → sin DLQ, sin commit, Worker se detiene | Sí |
| HA / multi-AZ productivo | No |
| TLS / WAF / autoscaling productivo | No |

## 8. Limitaciones conocidas

- At-least-once, **no** exactly-once E2E.
- AWS orientado a desarrollo/demostración, **no** HA.
- Sin TLS, WAF ni autoscaling productivo.
- Mobile sin publicación en tiendas de aplicaciones.
- OpenTelemetry sin collector, Grafana, Tempo ni Prometheus incluidos.
- Druid productivo no desplegado (analytics mockeado / Timescale).
- OAuth2 / mTLS completos no implementados.
- Enrolamiento demo de dispositivos solo en Development con flag explícito.

## 9. Procedimiento de actualización

1. Respaldar volúmenes/datos persistentes (TimescaleDB) antes de actualizar.
2. Obtener el tag y desplegar:

```bash
git fetch --tags
git checkout v1.0.0
docker compose --profile app up -d --build
```

3. Verificar health checks y smoke (`scripts/smoke-test.sh` o `.ps1`).
4. Revisar migraciones de esquema (`docs/database-migrations.md`); el
   `DatabaseInitializer` es idempotente, pero un backup es obligatorio si hay
   datos de valor.

## 10. Procedimiento de rollback

1. Si el despliegue falla **sin** incompatibilidad de esquema irreversible:
   volver al commit/tag anterior conocido de `main` y recrear el stack Compose.

```bash
git fetch --tags origin
git checkout <tag-o-sha-anterior-conocido>
docker compose --profile app up -d --build
```

2. Si hubo cambio de esquema incompatible: restaurar el respaldo de TimescaleDB
   tomado antes de la actualización y luego volver al tag anterior.
3. **No existe** un tag semántico anterior a `v1.0.0` en este repositorio; el
   rollback apunta al SHA estable previo en `main` (p. ej. el merge commit
   anterior al release).
